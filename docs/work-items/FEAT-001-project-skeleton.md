## 2. User Story

**As a** solo tech lead driving feature delivery (see `docs/personas/primary-user.md`), **I want to** run `orchestrator serve`, `orchestrator doctor`, and the migration/test commands against a fresh clone and see them all succeed on an empty project, **so that** every subsequent feature has a stable, conventional shell to extend instead of relitigating skeleton choices per PR.

---

## 3. Goal

A fresh clone of this repo builds, starts, passes health checks, and runs its test suite against a real Postgres — with no agent logic yet, but every seam the later features need (FastAPI app, Typer CLI, Alembic, LLM abstraction factory, HTTP client config, webhook receiver skeleton, RFC 7807 error handler, structured logging, Docker Compose) in place and conforming to the `python-ai-agent-service-docker-compose` profile.

---

## 4. Feature Scope

### 4.1 Included

- `pyproject.toml` with project metadata, `[project.scripts] orchestrator = "app.cli:main"`, pinned dependencies, `[tool.orchestrator]` config table, and dev/test dependency groups.
- Directory layout exactly as documented in `CLAUDE.md` → "Key Directories" and the stack profile's solution structure.
- FastAPI app (`src/app/main.py`) with router registration, global exception handler producing RFC 7807 Problem Details, `{ data, meta? }` response envelope helper, and structured JSON logging with `run_id` / `step_id` contextvars (no-op values in v1 — the plumbing is what matters).
- Typer CLI (`src/app/cli.py`) exposing the full command surface from `docs/ui-specification.md` — each command wired to a stubbed service function that returns a clearly-marked "not implemented yet" result. `doctor` and `serve` are fully implemented.
- Pydantic-settings config (`src/app/config.py`) reading env vars + `pyproject.toml [tool.orchestrator]` with precedence per the UI spec, validating at startup, and exposing typed accessors.
- Core infrastructure: `core/database.py` (AsyncEngine + async_sessionmaker + Base), `core/dependencies.py` (get_db_session, get_api_key), `core/exceptions.py` (AppError hierarchy + global handler), `core/llm.py` (provider-agnostic factory returning a stub provider by default for tests).
- `ai` module scaffold: empty `router.py` registered under `/api/v1` and `/hooks/engine`, `service.py` with function signatures matching the data model + API spec but `NotImplementedError` bodies, `models.py` with all five SQLAlchemy entities (`Run`, `Step`, `PolicyCall`, `WebhookEvent`, `RunMemory`) per `docs/data-model.md`, `schemas.py` with Pydantic DTOs matching `docs/api-spec.md`, `dependencies.py`, and an empty `tools/` package.
- Alembic initialized with async env template; one initial autogenerated migration creating all five tables + indexes + enum check constraints per the data model.
- HMAC webhook signature verifier (`core/webhook_auth.py` or similar) — real implementation, not a stub — plus a Bearer-token API-key dependency for control-plane routes. The webhook route skeleton uses the verifier and persists inbound events via the (stub) service even though no engine events will arrive yet.
- `GET /health` implementing the dependency-check chain from `api-spec.md` (DB ping, LLM provider liveness, flow-engine reachability; the latter two are best-effort and green-by-default if unconfigured, so `doctor` can still pass on a fresh clone).
- `Dockerfile` (multi-stage, python:slim final), `.dockerignore`, `docker-compose.yml` (Postgres only), `docker-compose.prod.yml` (API container on shared `infra` network), `.env.example`, `.env.production.example`.
- Test scaffold: `tests/conftest.py` with async fixtures (test DB per session, `AsyncClient`, HMAC signing helper, `StubPolicy`), one smoke test per module (`test_health.py`, `test_cli_doctor.py`, `test_migrations_apply.py`), and `tests/contract/` directory with a `@pytest.mark.live`-gated placeholder.
- Dev-tooling config: `pyproject.toml` sections or sibling files for `ruff` (lint + format), `pyright` (strict), and `pytest` (asyncio strict mode, markers for `live`).
- README section documenting the commands from `CLAUDE.md` → "Common Commands" (keep it short; link to the docs).

### 4.2 Excluded

- Any agent loop logic. The runtime, policy calls, stop conditions, and memory handling ship in later features.
- Tool implementations beyond the empty `tools/` package. No `generate_tasks`, `generate_plan`, etc., yet.
- Flow-engine client beyond a typed `httpx` wrapper class with configured base URL and auth header. No real node-dispatch methods; those land with the runtime loop feature.
- Any LLM provider adapter beyond the stub. Anthropic wiring ships with the first feature that actually needs a policy call.
- JSONL trace writer. The `TraceStore` protocol is defined and a no-op implementation is wired, but the JSONL writer ships with the runtime loop.
- Celery, Redis, pgvector, background workers. Out of scope per profile (optional) and out of scope per v1 stakeholder lock.
- Authentication beyond single API-key Bearer + webhook HMAC. No JWT, no multi-user, no roles.
- Any CI configuration beyond making `uv run pytest`, `uv run pyright`, and `uv run ruff check .` pass locally. GitHub Actions wiring is a separate IMP.

## 5. Acceptance Criteria

- **AC-1**: From a fresh clone, `uv sync && docker compose up -d && uv run alembic upgrade head && uv run orchestrator doctor` exits `0` with every check marked ✓.
- **AC-2**: `uv run orchestrator serve` starts the FastAPI app; `curl localhost:8000/health` returns `200` with a `{ data: { status: "ok", checks: {...} } }` envelope; `/docs` renders OpenAPI for all endpoints in `api-spec.md`.
- **AC-3**: `uv run pytest` runs green on a machine with Docker running. The suite includes the composition-integrity smoke test placeholder (asserts the runtime module imports and the `StubPolicy` is wired, even though the real loop doesn't exist yet).
- **AC-4**: `uv run pyright` and `uv run ruff check .` both pass with zero errors and zero warnings.
- **AC-5**: Alembic migration `upgrade head` + `downgrade base` + `upgrade head` round-trips cleanly on a fresh database.
- **AC-6**: Every endpoint in `api-spec.md` responds at the documented path — the control-plane endpoints return `501 Not Implemented` (as RFC 7807 Problem Details) from a stub service; the webhook endpoint fully accepts and persists valid HMAC-signed events and rejects unsigned ones with `401` while still persisting them with `signature_ok=false`.
- **AC-7**: Every CLI command in `ui-specification.md` is invocable with `--help` producing the expected synopsis; `doctor` and `serve` work end-to-end; all other commands return a clearly-marked "not implemented" result with exit code `2`.
- **AC-8**: `docker compose -f docker-compose.prod.yml build` succeeds. Running the built API image with a compose-supplied Postgres produces the same `/health` response as local dev.
- **AC-9**: No business logic is present in `router.py` or `cli.py`. Each route/command delegates to a service function (even if that service function raises `NotImplementedError`). This is enforced by a `tests/test_adapters_are_thin.py` AST-level check.
- **AC-10**: Every file referenced from `CLAUDE.md` → "Key Directories" exists, and no files outside that tree are introduced without a matching update to the doc.

## 9. Edge Cases

- Fresh clone with no `.env`: `doctor` MUST explain exactly which env vars are missing and exit `2`. It MUST NOT crash with a raw `pydantic.ValidationError`.
- Alembic autogenerate producing a migration that includes tables from SQLAlchemy metadata not in our models (e.g., `alembic_version` alone is fine; unexpected extras indicate misconfigured `target_metadata`). CI/test must fail loudly if this happens.
- Webhook endpoint receiving a request with a valid signature but unknown `engineRunId`: MUST return `404`, persist the event, and not crash the service.
- Webhook endpoint receiving a second event with the same `dedupe_key`: MUST return `202` idempotently without inserting a duplicate row.
- CLI run without `ORCHESTRATOR_API_KEY` set AND without `--api-base` reachable: MUST exit `2` with a clear "configure API key or start the service" message.
- Docker build running on a machine without Postgres locally: `docker compose up -d` must start Postgres from the compose file, not depend on a host install.
- Running `alembic upgrade head` twice back-to-back must be a no-op (no duplicate-table errors).
- Structured logging on a request that never populates `run_id`: contextvars MUST default to `None` and the JSON formatter MUST omit the field rather than emitting `"run_id": null`.

## 10. Constraints

- MUST adopt the `python-ai-agent-service-docker-compose` profile as-is. Deviations require an ADR update in the shared architecture repo, not a local workaround.
- MUST NOT modify `carestechs-flow-engine` — the orchestrator adapts to the engine's API, per AD-1.
- MUST NOT introduce Celery, Redis, pgvector, a frontend, or any LLM provider SDK in the composition root beyond the stub. Those come with the features that need them.
- MUST keep the CLI as a client of the HTTP service (via a shared service layer), not a back door to the DB — per CLAUDE.md anti-patterns.
- MUST target Python 3.12+ and use `uv` as the package manager per the profile.
- The work is a single PR if practical, or a stacked series in strict dependency order. Never merge a partial skeleton that leaves `doctor` red.

## 11. Motivation and Priority Justification

**Motivation:** Every subsequent feature (runtime loop, policy via tool calling, JSONL trace, first lifecycle agent) assumes a running FastAPI service, applied migrations, and a Typer CLI wired to the same services. Building them without the skeleton means duplicating setup decisions inside feature PRs — exactly the routing-in-your-head anti-pattern the orchestrator exists to eliminate. Doing this as FEAT-001 also lets us exercise the *framework* once before we try to run it *through itself*.

**Impact if delayed:** Every other work item is blocked. The "eat our own dog food" success metric (run at least one feature of this repo end-to-end through the orchestrator) cannot begin until the runtime loop can start, and the runtime loop can't start without the skeleton.

**Dependencies on this feature:** All in-flight orchestrator work. In particular, the runtime loop + JSONL trace + first lifecycle agent are direct successors.
