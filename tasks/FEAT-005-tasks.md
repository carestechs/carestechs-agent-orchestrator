**Technical Notes:**

**Type:** Database
**Workflow:** standard
**Complexity:** M
**Dependencies:** None

**Description:**
Add a new `RunSignal` entity to persist operator-injected signals (v1: `implementation-complete`; future: siblings). **Acceptance Criteria:**

- [ ] `dedupe_key` UNIQUE constraint — named `uq_run_signals_dedupe_key` per project convention.
- [ ] BTREE index on `(run_id, received_at)` — named `ix_run_signals_run_id_received_at`.
- [ ] Alembic migration file generated with `uv run alembic revision --autogenerate -m "add run_signals"`; the file's `upgrade` and `downgrade` are both reversible and round-trip clean on a local Postgres.
- [ ] `RunSignalDto` Pydantic model with camelCase aliases: `id, runId, name, taskId, payload, receivedAt, dedupeKey`.
- [ ] No change to the existing `WebhookEvent` table — signals are a separate concern per §6 "Alternative considered" in the brief.
- [ ] `uv run pyright`, `uv run ruff check .`, `uv run pytest tests/modules/ai/test_models.py` all green.
- [ ] Data-model doc updated (T-106 covers the full sweep — here: just add the `RunSignal` section; no changelog entry yet, that's in T-106).

**Files to Modify/Create:**
- `src/app/modules/ai/models.py` — new `RunSignal` SQLAlchemy class.
- `src/app/modules/ai/schemas.py` — new `RunSignalDto` Pydantic model + `SignalCreateRequest` for the endpoint body.
- `src/app/modules/ai/repository.py` — one new function `create_run_signal(session, run_id, name, task_id, payload, dedupe_key)` with ON CONFLICT DO NOTHING semantics (returns the existing row when dedupe_key collides).
- `src/app/migrations/versions/2026_04_18_add_run_signals.py` — new Alembic migration.

**Technical Notes:**
`dedupe_key = hashlib.sha256(f"{run_id}:{name}:{task_id or ''}".encode()).hexdigest()` — deterministic, collision-safe within this entity's scope. Persist the computed key; do NOT re-derive at read time. For the repository's ON CONFLICT path: use `insert(...).on_conflict_do_nothing(index_elements=["dedupe_key"]).returning(RunSignal)` so the repository returns `None` on duplicate (or re-SELECT to return the existing row — the endpoint T-098 wants the existing row for idempotency response). The migration MUST NOT autogenerate `ALTER TABLE` on existing tables; if the diff shows collateral changes from an unrelated model drift, back them out. Test the migration locally with `uv run alembic upgrade head && uv run alembic downgrade base && uv run alembic upgrade head`.

- [ ] `from_run_memory(run_memory)` — accepts `RunMemory.data` (a `dict`) and returns a validated `LifecycleMemory`; an empty dict yields `empty()`.
- [ ] `to_run_memory(lm)` — returns a plain `dict` (not a Pydantic model) suitable for assignment to `RunMemory.data`.
- [ ] Round-trip test: `to_run_memory(from_run_memory(to_run_memory(lm)))` is byte-identical to `to_run_memory(lm)`.
- [ ] `uv run pyright` clean — no `Any` escapes in the memory API.

**Files to Modify/Create:**

**Technical Notes:**
Use `model_config = ConfigDict(populate_by_name=True, alias_generator=to_camel, extra="forbid")` — `extra="forbid"` catches typos in tool writers before they corrupt the shape at runtime. `LifecycleTask.status` is a string literal type `"pending" | "in_progress" | "completed" | "failed"` (separate from `StepStatus` — this is per-task lifecycle state, not per-step). Don't reuse `StepStatus` even though the values look similar; coupling those enums will force a schema-change dance later when task lifecycle grows a state the step doesn't have (e.g., `awaiting_implementation_signal`).

### T-091: `generate_tasks` tool

**Type:** Backend
**Workflow:** standard
**Complexity:** S
**Dependencies:** T-089

**Description:**
Implement `src/app/modules/ai/tools/lifecycle/generate_tasks.py`: tool that takes `{"work_item_id": "...", "tasks_markdown": "<full document the LLM rendered>"}`, writes it atomically to `tasks/<work_item_id>-tasks.md` via temp-file-rename, and populates `memory.tasks` by parsing back the written file's task list. Refuses to overwrite an existing target file.

**Rationale:**
The task_generation stage's whole job is to produce this file. Atomic write means a process crash mid-render can't leave a half-baked file that blocks re-run. Refuse-to-overwrite is the v1 contract for "no resume after partial completion" (§4.2, excluded from scope).

**Acceptance Criteria:**
- [ ] Writes to `tasks/<work_item_id>-tasks.md` (path joined against `Settings.repo_root`).
- [ ] Temp-file-rename: writes to `tasks/.<work_item_id>-tasks.md.tmp.<pid>.<uuid>`, fsyncs, then `os.rename` to final path.
- [ ] Raises `PolicyError("tasks file already exists: <path>")` when the target exists (even a zero-byte file).
- [ ] Parses back the written file's task list (same regex as `load_work_item` uses on the identity table, adapted for task rows — captures T-XXX ids + titles) and populates `memory.tasks`.
- [ ] Empty `tasks_markdown` or a rendered doc with zero parseable T-rows raises `PolicyError("generated task list is empty")`.
- [ ] Unit tests: happy path, refuse-to-overwrite, empty-body, file-path-escape-root refusal.
- [ ] Temp file is cleaned up on any exception (including OS-level write failure).

**Files to Modify/Create:**
- `src/app/modules/ai/tools/lifecycle/generate_tasks.py` — tool adapter.
- `src/app/modules/ai/tools/lifecycle/atomic_write.py` — shared `write_atomic(path, content)` helper (T-093, T-094, T-095 also use it).
- `tests/modules/ai/tools/lifecycle/test_generate_tasks.py` — 4 cases.
- `tests/modules/ai/tools/lifecycle/test_atomic_write.py` — 3 cases: happy, temp-file-leftover-cleaned-on-exception, path-escape-root refusal.

**Technical Notes:**
`write_atomic` is the *one* place that touches `tempfile` + `os.rename` + `os.fsync`. Every other tool that writes a file goes through it — simpler review, easier to harden later (e.g., directory fsync for crash-consistency). Use `os.open(path, O_WRONLY | O_CREAT | O_EXCL)` instead of `open(..., "x")` so the exclusivity check is at the OS level (no TOCTOU between Python-side exists-check and write). The task-list parser reuses the same regex machinery as `parse_work_item` — refactor into a shared module when T-094 also needs it (it doesn't — planning diffs work files, not task rows, so don't refactor prematurely).

### T-094: `review_implementation` tool + `git.py` diff boundary

**Type:** Backend
**Workflow:** standard
**Complexity:** M
**Dependencies:** T-089, T-091 (for atomic writer)

**Description:**
Implement two pieces in the same PR:
1. `src/app/modules/ai/tools/lifecycle/git.py` — `get_diff(paths: list[str], base: str = "main") -> str` using `subprocess.run(["git", "diff", f"{base}...HEAD", "--", *paths], check=True, text=True, timeout=10)`. Quarantined import of `subprocess` (no other module may import it — enforced by T-106's adapter-thin extension). Wraps every failure in `PolicyError`: missing git binary, not a repo, detached HEAD, non-zero diff exit, timeout.
2. `src/app/modules/ai/tools/lifecycle/review_implementation.py` — tool that takes `{"task_id": "T-XXX", "verdict": "pass" | "fail", "feedback": "..."}`, writes to `plans/plan-<task_id>-<slug>-review-<attempt>.md` (attempt = `len(memory.review_history[task_id]) + 1`), and appends a `LifecycleReview` to `memory.review_history`. The `git diff` for the task's files-touched list is NOT fetched by this tool — that's the runtime's job when it builds the prompt (see T-100 for how the lifecycle-agent YAML wires this). This tool only records the verdict.

**Rationale:**
Separation of concerns: `git.py` is the *only* subprocess boundary in the codebase (everything else is async I/O). Keeping it narrow means the adapter-thin test can forbid `subprocess` everywhere else. `review_implementation` itself is a file-writer — it doesn't run `git`, which keeps it testable without a git repo on the test machine.

**Acceptance Criteria:**
- [ ] `get_diff` returns the stdout of `git diff` on the happy path, trimmed to 64 KB with a truncation marker `\n...<truncated N bytes>...\n` appended.
- [ ] Timeout, missing binary, non-zero exit, no-repo, detached HEAD — each surface as `PolicyError` with a distinct message.
- [ ] Review tool writes to `plans/plan-<task_id>-<slug>-review-<attempt>.md`; attempt is 1-based and derives from `len(memory.review_history)` filtered to this task.
- [ ] `verdict` accepts only `"pass"` or `"fail"`; anything else → `PolicyError("invalid review verdict: <v>")`.
- [ ] Appends one `LifecycleReview` entry with `{task_id, attempt, verdict, feedback, written_to}` to memory.
- [ ] Unit tests for `get_diff`: happy (uses a temp git repo fixture), timeout, missing binary (monkeypatch PATH), non-zero exit. For `review_implementation`: happy pass, happy fail, invalid verdict, writes-to-correct-attempt-filename, memory append.

**Files to Modify/Create:**
- `src/app/modules/ai/tools/lifecycle/git.py` — only subprocess import in the codebase.
- `src/app/modules/ai/tools/lifecycle/review_implementation.py` — tool adapter.
- `tests/modules/ai/tools/lifecycle/test_git.py` — 4 cases; uses a `tmp_path` + `subprocess.run(["git", "init", ...])` fixture for the happy case.
- `tests/modules/ai/tools/lifecycle/test_review_implementation.py` — 5 cases.

**Technical Notes:**
`subprocess.run(shell=False)` always — argv list, never a string. `timeout=10` is generous for a diff; if a real diff ever takes longer, it probably means the task is too big. Truncation: `diff[:65536] + trailing_marker if len(diff) > 65536 else diff`. The 64 KB limit is deliberately conservative — 64 KB of diff is already hard for a human to review, let alone an LLM. For the `test_git.py` happy case, build a tiny temp repo in `tmp_path`, `git init`, commit a file, branch + modify + commit again, then assert `get_diff(["file"], base="main")` returns the second-commit's diff. This test is the only one in the suite that actually shells out; mark it `@pytest.mark.requires_git` if the CI image might not have git (the Dockerfile already installs it — verify).

### T-096: `wait_for_implementation` tool + `PauseForSignal` sentinel + runtime pause handling

**Type:** Backend
**Workflow:** standard
**Complexity:** L

**Description:**

- [ ] `supervisor.await_signal` blocks on an `asyncio.Event` keyed by `(run_id, name, task_id)` — distinct events; signaling one task's completion does NOT wake another task's wait.

### T-097: Correction-bound enforcement + `stop_reason=error` with `final_state.reason`

**Type:** Backend
**Workflow:** standard
**Complexity:** M
**Dependencies:** T-089, T-096

**Description:**
Add the per-task correction counter and enforcement path. Three pieces:
1. `corrections` node (purely a policy-decision point — no new tool) increments `memory.correction_attempts[task_id]` on entry via a new runtime hook (`on_node_entry: (node_name, memory) -> memory`) — add to `src/app/modules/ai/runtime.py`.
2. `src/app/modules/ai/stop_conditions.py` — new `correction_budget_exceeded(memory, max_corrections)` pure function returning `StopReason.error | None`. Priority: same as `error` (inserted before `error` so the more-specific reason wins the trace).
3. Runtime loop consults `correction_budget_exceeded` after every iteration; on trip, terminates with `stop_reason=error` and writes `final_state = {"reason": "correction_budget_exceeded", "task_id": "...", "attempts": N}`.
Setting: `LIFECYCLE_MAX_CORRECTIONS` env-var, default `2`, read via `Settings`.

**Rationale:**
AC-6. Without the bound, a flaky policy or a stubborn bug can infinitely loop between `implementation` → `review` → `corrections`. Two attempts is enough for the normal "small fix" recovery; three attempts means the task is too big or the policy doesn't understand the feedback — both are out-of-band problems.

**Acceptance Criteria:**
- [ ] `Settings.lifecycle_max_corrections: int = 2` with env override `LIFECYCLE_MAX_CORRECTIONS`.
- [ ] `correction_budget_exceeded(memory, max)` returns `StopReason.error` when any task's attempts > max; else `None`.
- [ ] The stop-conditions priority order per CLAUDE.md ("cancelled > error > budget_exceeded > policy_terminated > done_node") still holds — the correction-bound trip fires under the `error` priority bucket, not a new level.
- [ ] `final_state.reason == "correction_budget_exceeded"`, `final_state.task_id` and `final_state.attempts` are populated.
- [ ] `on_node_entry` hook is called once per entry into `corrections` — not per policy call within that node.
- [ ] Unit tests: counter increments correctly, threshold trip returns `error`, priority ordering respects existing error bucket.
- [ ] Integration test: drive a stub-policy run that loops review → corrections → implementation three times; assert the run terminates with `stop_reason=error` + `final_state.reason=correction_budget_exceeded`.

**Files to Modify/Create:**
- `src/app/config.py` — add `lifecycle_max_corrections` field.
- `src/app/modules/ai/stop_conditions.py` — new `correction_budget_exceeded` function; extend `evaluate` to consult it.
- `src/app/modules/ai/runtime.py` — add `on_node_entry` hook + call to it at each loop iteration; write `final_state` with reason/task_id/attempts on trip.
- `tests/modules/ai/test_stop_conditions.py` — extend with 3 new cases.
- `tests/integration/test_runtime_corrections.py` — 1 case: stub-policy loop that exceeds the bound.

**Technical Notes:**
The `on_node_entry` hook is a simple mechanism — a single function called from the runtime after the policy has selected the node but before the tool is invoked. Don't build a generic pluggable hook system; one call site, one function. The increment on entering `corrections` (not on entering `implementation`) is deliberate: the operator's first implementation attempt isn't a "correction." Only after review fails and the flow loops back does the corrections counter tick. This matches operator intuition and the AC-6 wording.

### Risks / Open Questions

- **Anthropic-mock fixture drift** (T-102). The 8 stage fixtures are frozen snapshots of real Claude responses. If the Anthropic API's response shape changes (new required fields, renamed tool-call fields), the fixtures go stale silently — the test will pass but the real API will break. Mitigation: T-104's `@pytest.mark.live` contract test is the drift detector. Run it as part of the FEAT-005 ship PR; optionally add a nightly scheduled job to `--run-live` against `main` after ship.
- **Git diff size blowup** (T-094). 64 KB truncation is a guess. A real review of a non-trivial implementation can genuinely exceed that if Claude wrote a large refactor. If the truncation marker becomes a common appearance in traces, raise the ceiling or split reviews by file. Don't fix in v1 — wait for evidence.
- **Operator error: signal wrong task_id** (T-098). AC-5 covers 404 for the "task doesn't exist" path, but an operator who marks `T-002` done when they meant `T-001` lands a signal against the wrong task. The agent then advances to review on the wrong task's git diff, likely gets a confused `fail` verdict, and burns a correction attempt. Low-probability in practice (operators track what they're implementing), but a future FEAT-006 confirmation-prompt CLI flag (`--confirm`) would harden it.
- **Prompt-routing edge: mixed work-item type** (T-100). The YAML maps `task_generation` to `feature-tasks.md` as default; runtime substitutes by type. What if a FEAT brief is reclassified to a BUG mid-lifecycle? Won't happen in practice (work-item type is set at creation and never changes), but the runtime doesn't assert the type stays constant across a run. Document in the investigation plan; pin the type at intake; if it changes, fail loudly.
- **Zombie runs and signal replay** (T-098 + T-096). A run that was suspended at implementation when the orchestrator process died gets flipped to `failed` by the FEAT-002 zombie reconciliation on startup. But what about any signal the operator submitted during the downtime? It lands in `run_signals` (persisted), but the supervisor never sees a wake because the run is already terminal. The operator's signal is preserved for forensics but not delivered. Acceptable for v1 (the operator realizes the run died and re-runs with a fresh work item); document in CLAUDE.md + the stakeholder-definition FAQ.
- **Correction-bound false trips under flaky policy** (T-097). `LIFECYCLE_MAX_CORRECTIONS=2` is aggressive for a flaky first real agent. If operators routinely hit the bound on legitimate tasks, raise it (env var, no code change). If the bound is still hitting, the problem isn't the bound — it's either the review prompt or the policy's inability to follow feedback, and that's a prompt-engineering concern for a follow-up.
- **`agent_definition_hash` determinism** (T-100). YAML canonicalization already passes through JSON with sorted keys, but Path objects serialize differently across OSes. Verify the hash is identical on Linux / macOS / Windows CI; if not, normalize Path values to POSIX strings before hashing.
